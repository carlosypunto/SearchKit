Retrieval-Augmented Generation grounds a language model in your data. Instead of hoping the model memorized the answer, you retrieve the relevant passages at question time and hand them over as context. The generator's job shrinks from "know everything" to "read these paragraphs and answer" — dramatically reducing hallucination and unlocking private or fresh knowledge.

The architecture splits into an offline and an online path. Offline, ingestion runs per document: split text into chunks (paragraph-scale, with overlap so ideas aren't severed at boundaries), embed each chunk into a vector, and persist vectors alongside the source text and metadata in a vector store. Online, each query is embedded with the same model, nearest-neighbor search returns the top-K chunks, and a prompt template assembles them with the question and grounding instructions.

Pure semantic retrieval has a known blind spot: exact identifiers, product names, and rare terms that embeddings blur. Hybrid search fixes it by running lexical retrieval (BM25) in parallel and fusing the two rankings — Reciprocal Rank Fusion needs no score calibration and works remarkably well. Downstream refinements include reranking the candidate set with a cross-encoder and deterministic rules that guarantee must-have results, like exact title matches.

An entirely on-device incarnation is now practical on Apple platforms: contextual embeddings from the NaturalLanguage framework, sqlite-vec for vector KNN inside SQLite, FTS5 for the lexical leg, and RRF to fuse. Retrieval latency stays in single-digit milliseconds for catalogs of thousands of chunks, and no document ever leaves the device.

Two failure modes account for most disappointing RAG systems. First, weak retrieval blamed on the generator — evaluate retrieval in isolation with known query-answer pairs before touching prompts. Second, unversioned vector spaces: embeddings from different models or revisions are mutually meaningless, so persist a manifest describing model, dimension, pooling, and transforms, and rebuild the index when any of it changes.

Ingestion deserves more engineering attention than it usually receives, because it is the part of the pipeline that runs repeatedly as the corpus evolves. Incremental sync is the baseline requirement: hash each document's raw content, compare against the hashes recorded at the last indexing pass, and re-embed only documents whose hash changed. Documents that disappeared from the source must be removed from the index, and unchanged documents should cost nothing. Without this diff, every content update triggers a full re-embed, and full re-embeds are exactly the kind of multi-minute job that teams start postponing until the index silently drifts out of date.

Deletion is subtler than insertion. When a document shrinks from five chunks to three, replacing the first three rows is not enough — the two stale chunks keep matching queries with text that no longer exists in the source. The robust pattern is delete-then-insert per document: remove every row belonging to the document, insert the new chunk set, and do both inside one transaction so a crash cannot leave the document half-indexed. Deterministic chunk identifiers derived from document ID and ordinal make this idempotent; re-running ingestion after a failure converges to the same state instead of accumulating duplicates.

Prompt assembly is the stage everyone underestimates. The retrieved chunks must be formatted so the model can cite them: prefix each with its source title and position, separate them with unambiguous delimiters, and state the grounding rule explicitly — answer only from the context, and say so when the context does not contain the answer. Order matters too: models attend more reliably to the beginning and end of the context window, so place the strongest candidates first. Budget tokens deliberately; ten mediocre chunks crowd out the two that matter, and a smaller, cleaner context routinely beats a larger, noisier one.

Evaluation separates RAG systems that improve from ones that merely change. Build a golden set of a few dozen questions with known source passages, then measure two things independently: does retrieval return a passage containing the answer within the top K (hit rate, mean reciprocal rank), and does the generator produce a faithful answer when it does? The split matters because the fixes differ — retrieval misses call for better chunking, hybrid fusion, or a different embedding model, while generation failures call for prompt changes. A retrieval evaluation runs in seconds without any LLM, so it can gate every change in continuous integration.

Operationally, treat the vector index as a disposable cache, never as a system of record. The source documents plus the manifest fully determine the index, so rebuilding must always be safe, automatic, and idempotent. This principle simplifies every hard case: schema migrations become "wipe and rebuild", corrupted files become "wipe and rebuild", and model upgrades become a manifest mismatch followed by — again — "wipe and rebuild". Systems that instead try to migrate vectors between incompatible spaces accumulate subtle relevance bugs that no one can debug, because the individual numbers look plausible while the geometry is meaningless.
